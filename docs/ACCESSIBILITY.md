# Accessibility beyond ARIA

This doc covers important accessibility topics **other than ARIA** that affect this site. ARIA is already used for roles, labels, and live regions; these areas complement that.

---

## 1. **Keyboard navigation**

- **Tab order**: All interactive elements (links, buttons, inputs) should be reachable in a logical order with Tab. Avoid `tabindex` > 0 unless you have a specific reason (e.g. programmatic focus).
- **Enter / Space**: Buttons and links must activate with Enter; buttons (and button-like divs) also with Space. Custom clickable `<div>`s need `onKeyDown` for Enter/Space (you already do this on org expand/collapse).
- **Escape**: Modals should close on Escape (CalendarModal already does).
- **Focus trapping**: When a modal is open, focus should stay inside it until it closes. Consider adding a focus trap to CalendarModal for screen-reader and keyboard users.
- **Focus restoration**: When a modal closes, focus should return to the element that opened it.

---

## 2. **Visible focus**

- **Never remove focus outline without a replacement.** You use `focus:ring-2` (and similar) in many places—keep that pattern for all interactive elements.
- **Contrast**: Focus ring (e.g. teal/emerald) should be clearly visible on all backgrounds (including dark mode). Your `focus:ring-offset-2` and `dark:focus:ring-offset-slate-900` help.
- **Consistency**: Use the same focus style (ring color/width) for similar components so keyboard users learn the pattern.

---

## 3. **Color and contrast**

- **Text**: Aim for WCAG AA—4.5:1 for normal text, 3:1 for large text (e.g. 18px+ or 14px bold). Your slate/white and dark-mode palettes are generally safe; check any low-contrast combos (e.g. gray-400 on white).
- **Not color alone**: Don’t rely only on color to convey meaning (e.g. errors). You use both color and text/icons (e.g. “Loading…”, “Error”), which is good.
- **Links**: Ensure links are identifiable without color alone (underline, icon, or sufficient contrast). Your `hover:underline` and focus rings help.

---

## 4. **Motion (prefers-reduced-motion)**

- **Respect the user setting.** Some users are sensitive to motion (vestibular disorders, migraine).
- **Implemented**: In `globals.css`, `prefers-reduced-motion: reduce` is used to turn off the principal carousel animation and shorten other animations/transitions. Keep this in mind when adding new motion.
- **Best practice**: Prefer `prefers-reduced-motion` in CSS (or a hook that reads the media query) so animated components (carousels, toasts, page transitions) can disable or simplify motion.

---

## 5. **Semantic HTML**

- **Headings**: Use a single `<h1>` per page and a logical order (h1 → h2 → h3). Don’t skip levels. You use h1/h2/h3 appropriately.
- **Landmarks**: Use `<main>`, `<nav>`, `<header>`, `<footer>`, `<aside>`, `<section>` so screen readers can jump by region. You already do this and pair them with ARIA where needed.
- **Buttons vs links**: Use `<button>` for actions (submit, open modal, toggle); use `<a href="...">` for navigation. You follow this; the only clickable div (org row) has `role="button"` and keyboard handling.
- **Lists**: Use `<ul>`/`<ol>` for list content so assistive tech can announce the number of items. You use lists for search results, org details, etc.

---

## 6. **Images and media**

- **Alt text**: Every meaningful image needs descriptive `alt`. Decorative images should have `alt=""`. You use `alt={title}` on QuiltSquare images and `alt=""` where appropriate (e.g. org photo when the name is next to it).
- **Video**: The intro video has `aria-label="Intro video"`. Consider captions for any speech or critical audio (e.g. a future video with dialogue).
- **Autoplay**: Intro video is muted by default, which is good for autoplay policies and accessibility.

---

## 7. **Forms**

- **Labels**: Every form field should have a visible or screen-reader-only `<label>` associated via `htmlFor`/`id`. You do this consistently.
- **Errors**: Associate error messages with inputs (e.g. `aria-describedby` pointing to the error element and `aria-invalid` on the input when invalid). Consider adding this to forms that show validation errors.
- **Required**: Use `required` and/or `aria-required="true"` and indicate required fields in the label (e.g. “Name *”).

---

## 8. **Page title and language**

- **`<title>`**: Should be unique per page and describe the page (e.g. “Search | All of Us For All of Us”). Update the root layout metadata: it currently says “Create Next App” / “Generated by create next app”.
- **`lang`**: Root `<html lang="en">` is set; if you add content in another language, use `lang` on that element (e.g. `<span lang="es">`).

---

## 9. **Touch targets**

- **Size**: Interactive elements should be at least about 44×44 px for touch. Small icon-only buttons (e.g. calendar “view” in a list) may be below that; consider increasing tap area with padding while keeping the visual icon smaller.
- **Spacing**: Enough space between tappable elements so users don’t hit the wrong one. Your spacing is generally fine.

---

## 10. **Text and readability**

- **Zoom**: Avoid locking zoom (no `maximum-scale` in viewport meta). Your app doesn’t prevent zoom.
- **Line length**: Long lines are harder to read; your max-width containers help.
- **Font size**: Prefer relative units (rem/em) or responsive sizing so users can scale text. Tailwind’s text scale is relative by default.

---
